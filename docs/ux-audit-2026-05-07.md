# Sub-Accounts UX Audit — 2026-05-07

**Subject:** `/sub-accounts` preview page (`src/shells/ShellV1/locations/`)
**Method:** Live screenshots of `https://ghl-design-previews.vercel.app/sub-accounts` in four states (default / cards selected / bulk-actions dropdown open / search empty state) evaluated against Nielsen's 10 heuristics and Gestalt's seven grouping principles.
**Severity legend:** P1 = must-fix (broken or confusing for first-time use). P2 = should-fix (friction-causing). P3 = nice-to-have (document, don't fix this pass).

> Layout-restructure findings (sticky sidebar, adaptive topbar) are tracked in Part 1 of the work order; everything in this audit is page-level UX.

---

## Findings

### [N1] Visibility of system status — selection count is invisible until the user re-reads the dropdown label
**Severity:** P1
**Where:** `BulkActionsRow.vue`
**Observation:** When 1+ cards are selected, the only count indicator on the page is baked into the Bulk Actions button's label (`Bulk Actions (2)`). The user has to look at a button on the far right of the row to discover what they've selected. There is no prominent "X selected" status text in the bulk-actions row, and the select-all checkbox going to indeterminate state is too subtle.
**Recommendation:** When `selectedCount > 0`, show a visible "**N selected**" text right next to (or in place of) the Select-All checkbox. Add a "Clear" link button next to it.
**Implemented:** ✅ in `BulkActionsRow.vue` — added "N selected" pill and "Clear" link visible only when `selectedCount > 0`.

---

### [N3] User control and freedom — no way to clear a selection in one click
**Severity:** P1
**Where:** `BulkActionsRow.vue`
**Observation:** Once cards are selected, the only way to clear them is to uncheck each card individually OR toggle Select-All twice. There is no obvious exit path from "I accidentally selected 5 cards."
**Recommendation:** Add a "Clear" link in the bulk-actions row that fires `clearSelection()`.
**Implemented:** ✅ in `BulkActionsRow.vue` + new `clear-selection` event wired in `Locations.vue`.

---

### [N5] Error prevention — destructive bulk action looks identical to non-destructive ones in the dropdown
**Severity:** P1
**Where:** `BulkActionsRow.vue` bulk dropdown options
**Observation:** In the open Bulk Actions dropdown, "Delete" sits between "Resume" and "Export to CSV" with the same neutral styling — no red color, no icon, no separator above. A user scanning the menu has no visual cue that Delete is irreversible. The confirmation modal does turn red, but only AFTER the menu pick.
**Recommendation:** In the dropdown options, give the Delete row an `error` color treatment (red text). Keep the divider before "Export to CSV" so the row visually anchors as "destructive group" → "data group".
**Implemented:** ✅ in `BulkActionsRow.vue` — added `props.style: { color: ... }` red treatment to the Delete option via the ghl-ui dropdown options API.

---

### [N5] Error prevention — destructive confirmation body is generic, doesn't name what's being deleted
**Severity:** P1
**Where:** `BulkActionConfirmModal.vue`
**Observation:** The current confirmation body for bulk delete is "This will affect every selected sub-account. Continue with this action?" — same string regardless of action type. For a destructive action, a user needs to see what will be lost ("X sub-accounts will be permanently deleted. This cannot be undone.") so the gravity of the choice is clear.
**Recommendation:** When `destructive` is true, swap the body to the stronger destructive copy that names the count and explicitly says it can't be undone.
**Implemented:** ✅ in `BulkActionConfirmModal.vue` + new i18n keys `confirmDestructiveBulkBody` / `confirmDestructiveCardBody`.

---

### [N9] Help users recover from errors — empty state has no recovery CTA
**Severity:** P1
**Where:** `Locations.vue` empty-state branch
**Observation:** When the search yields no matches, the page renders "No sub-accounts match your filters" with a magnifying glass icon and a sub-line, but no button to clear the search/filters. The user has to find their way back to the search input and clear it manually. This is recovery friction.
**Recommendation:** Add a primary "Clear filters" button inside `<UIEmpty>`. Clicking it should clear `searchText` AND reset `advancedFilters` AND reset `showAgencySubAccounts` to default.
**Implemented:** ✅ in `Locations.vue` — added `clearAllFilters()` + `<UIButton>` slotted into `<UIEmpty>` via the `#actions` slot. (Initial pass slotted into `#extra` which doesn't exist on `UIEmpty`; the button silently dropped. Verified the fix on live deploy: empty state now shows title, body, and primary CTA, and clicking Clear filters restores the full card list.)

---

### [N10] Help and documentation — icon-only buttons have no tooltips
**Severity:** P1
**Where:** `FilterStrip.vue`
**Observation:** The filter funnel, history clock, and view-toggle are icon-only buttons with no visible label and no tooltip. The history button uses the same icon shape twice (filter strip + bulk row) — the user can't tell what either one does without clicking.
**Recommendation:** Wrap each icon-only button in `<UITooltip>` with the action's name.
**Implemented:** ✅ in `FilterStrip.vue` — `UITooltip` added around funnel, history, and view-toggle buttons.

---

### [G1] Proximity — filter-strip controls are equidistant; no logical grouping
**Severity:** P2
**Where:** `FilterStrip.vue`
**Observation:** The strip renders eight controls (date, search, columns, sort-by, sort-direction, funnel, history, view-toggle) at uniform 8px gaps. Cognitively these belong to four sub-groups:
  1. **Scope** — date range + search
  2. **Display** — columns + sort + sort direction
  3. **Filter** — funnel
  4. **Utilities** — history + view toggle

The eye reads them as one undifferentiated row.
**Recommendation:** Bump the gap between groups (e.g. `gap-4` between groups, `gap-2` within groups) and add a vertical divider between Display and Filter, and between Filter and Utilities, so each group reads as one unit.
**Implemented:** ✅ in `FilterStrip.vue` — restructured into nested grouping divs with `filter-strip__group` + `filter-strip__divider` selectors.

---

### [G2 / N4] Similarity / Consistency — history icon appears twice, both open the same popover
**Severity:** P2
**Where:** `FilterStrip.vue` and `BulkActionsRow.vue`
**Observation:** The Bulk Action History popover is reachable from two icon-only buttons that look identical: one in the filter strip (between funnel and view toggle) and one in the bulk-actions row (between the show-agency switch and Bulk Actions). Two entry points = redundancy + a "what's the difference?" pause.
**Recommendation:** Keep history in the filter strip (it's a global utility, not a selection-dependent action). Remove the duplicate from the bulk-actions row.
**Implemented:** ✅ in `BulkActionsRow.vue` — duplicate history button removed.

---

### [N1] Visibility of system status — card selection border is too subtle on a busy page
**Severity:** P2
**Where:** `LocationCard97Plan.vue`
**Observation:** Selected cards get `border-color: var(--primary-500)` and a 4px halo. Visible, but on a page with multiple cards stacked tightly it's easy to miss the difference between selected and unselected at a glance. There is no left-edge accent, no checkmark badge, no row tint.
**Recommendation:** Add a 3px solid primary-500 left border in addition to the existing halo, so selected cards read as "marked" even at glance. Mirrors the GitHub PR file-list pattern.
**Implemented:** ✅ in `LocationCard97Plan.vue` — added the left accent.

---

### [N6] Recognition rather than recall — sort dropdown labels itself with the value, not the role
**Severity:** P2
**Where:** `FilterStrip.vue` Sort By button
**Observation:** The Sort By button shows just "Name" (the current value). A user looking at the strip can't tell whether "Name" is a label, a filter, or a sort target — they have to click to find out.
**Recommendation:** Prefix with the role: "Sort: Name". Same pattern many other GHL surfaces use.
**Implemented:** ✅ in `FilterStrip.vue` — changed `sortByLabel` computed to `Sort: {value}`.

---

### [N4] Consistency — sort-direction option labels disagree with the button label format
**Severity:** P2
**Where:** `FilterStrip.vue` sort-direction dropdown
**Observation:** Dropdown options say "Ascending (A-Z)" / "Descending (Z-A)" but the button shows just "A-Z" / "Z-A". The user picks a long-form label and gets back a short-form one — small consistency tax.
**Recommendation:** Change the button label format to also use the descriptive form ("A → Z" / "Z → A") so option-picked and button-displayed forms match closer.
**Implemented:** ✅ in `FilterStrip.vue` — `sortDirLabel` now returns `A → Z` / `Z → A` and dropdown option labels match.

---

### [G6] Common region — bulk-actions row mixes selection controls and global toggles in one strip with no visual region
**Severity:** P3
**Where:** `BulkActionsRow.vue`
**Observation:** The row contains four logical things: (1) select-all + selection status (left), (2) show-agency switch (right), (3) bulk-actions trigger (far right). They share one row but no visual region.
**Recommendation (deferred):** Wrap in a `<div>` with subtle background tint (`gray-50` deeper than the page bg) so the row reads as a "selection toolbar."
**Implemented:** ⏸ documented only.

### [G7] Alignment — date picker height (40px) doesn't match adjacent button height (40px) on some browsers
**Severity:** P3
**Where:** `FilterStrip.vue`
**Observation:** Visual alignment is generally fine, but the UIDatepicker renders slightly taller than UIButton on Safari builds, leaving a 1-2px baseline drift.
**Recommendation (deferred):** No fix this pass — sub-pixel issue specific to the ghl-ui datepicker primitive, not the page layout.
**Implemented:** ⏸ documented only.

### [N7] Flexibility and efficiency — no keyboard shortcut for select-all / clear-selection / bulk delete
**Severity:** P3
**Where:** Page level
**Observation:** Power users who manage hundreds of sub-accounts can't ⌘A to select all or Backspace to delete. No accelerator hints anywhere.
**Recommendation (deferred):** Out of scope for a stakeholder-review preview. If we ship this surface, wire keyboard handlers + add hint text near the bulk-actions row.
**Implemented:** ⏸ documented only.

---

### [N3] User control and freedom — confirm modal has no Esc-to-close hint
**Severity:** P3
**Where:** `BulkActionConfirmModal.vue`
**Observation:** Esc closes the modal (UIModal default), but the affordance isn't shown. Power users learn this; new users hit Cancel.
**Recommendation (deferred):** Subtle "Press Esc to cancel" footer text; not worth a P1/P2 trip.
**Implemented:** ⏸ documented only.
